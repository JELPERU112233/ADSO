01 ────
Host OS Kernel
Núcleo del Sistema Operativo

El kernel del host es el componente central que todos los contenedores comparten. A diferencia de las VMs, no hay un kernel separado por contenedor — todos corren sobre el mismo kernel Linux del host.

Provee las llamadas al sistema (syscalls) que los contenedores usan para acceder al hardware, gestionando memoria, scheduling de procesos y drivers de dispositivos de manera unificada.

02 ────
User Space Host
Procesos del espacio de usuario

Son los procesos normales del sistema operativo anfitrión que corren fuera de cualquier contenedor. Tienen acceso completo al sistema de archivos y recursos del host.

El daemon de Docker (dockerd) o LXD corre en este espacio. Actúa como orchestrador que crea y gestiona los contenedores llamando al kernel vía syscalls.

03 ────
Jailed Processes
Procesos enjaulados / Contenedores

Son grupos de procesos que el kernel confina en un "jail" (jaula) mediante namespaces. Cada contenedor cree ser el único proceso en el sistema — tiene su propio PID 1, su propia red, su propio sistema de archivos.

En la práctica: cuando corres docker run nginx, los procesos de nginx se ejecutan enjaulados. Ven su propio namespace de red (eth0 propio), su propio árbol de procesos (PID 1 = nginx), pero usan el kernel del host.

Múltiples contenedores pueden correr simultáneamente sin interferirse, tal como muestra el diagrama con 3 "Jailed processes" independientes.

04 ────
Namespaces
Aislamiento de recursos del kernel

Los namespaces son el mecanismo principal del kernel Linux para aislar la visibilidad de recursos. Cada contenedor tiene su propia "vista" del sistema:

PID — cada contenedor ve solo sus propios procesos (su init es PID 1)
NET — interfaz de red propia, tabla de rutas, firewall
MNT — sistema de archivos independiente (chroot avanzado)
UTS — hostname y domainname propios
IPC — comunicación interprocesos aislada
USER — mapeo de UIDs (root en contenedor ≠ root en host)

05 ────
Control Groups
cgroups — Limitación de recursos

Los cgroups (Control Groups) limitan y contabilizan los recursos que cada contenedor puede consumir. Si namespaces da "visibilidad aislada", cgroups da "recursos garantizados".

cpu — cuánto CPU puede usar el contenedor (ej: máx 50%)
memory — límite de RAM (ej: máx 512 MB, si supera → OOM kill)
blkio — velocidad de lectura/escritura en disco
net_cls — prioridad de tráfico de red

Ejemplo en Docker: docker run --memory=512m --cpus=1 nginx

06 ────
Hardware
Memory · I/O · CPUs

El hardware físico es compartido por todos los contenedores a través del kernel. A diferencia de las VMs donde el hipervisor emula hardware, aquí el acceso es directo y nativo.

Memory — RAM física gestionada por el kernel; cgroups limita cuánta usa cada contenedor
I/O — Disco y red; el kernel multiplexa el acceso entre contenedores
CPUs — El scheduler de Linux reparte tiempo de CPU entre todos los procesos (host + contenedores)

Esto explica por qué los contenedores tienen rendimiento casi nativo: no hay capa de emulación entre el proceso y el hardware.
